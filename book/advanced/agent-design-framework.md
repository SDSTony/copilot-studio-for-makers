# 에이전트 설계 프레임워크를 통한 에이전트 설계

## Agent Design Framework란?

Agent Design Framework는 에이전트의 목적, 트리거, 도구, 채널, 거버넌스 요구사항 등을 체계적으로 정의하도록 안내하는 **설계 사고 도구(thinking aid)** 입니다.

이 프레임워크는 딱딱한 템플릿이 아니라, 팀이 설계 결정을 정렬하고, 리스크를 조기에 발견하며, 흔한 함정을 피하는 데 도움을 줍니다.

### 언제 사용하면 좋은가?

모든 에이전트에 대해서 구축 전 설계를 해야 하는 것은 아닙니다. 하지만 **설계를 먼저 진행한다면 아래와 같은 장점**이 있습니다:

- ✅ 의존성 식별
- ✅ 필요한 데이터, 도구, 에이전트 플로우 사전 식별
- ✅ 보안 설계
- ✅ 명확한 평가 기준 선별

### 설계 프레임워크가 불필요한 상황

반대로 아래와 같은 경우에는 굳이 프레임워크를 사용하지 않아도 됩니다:

- 간단한 개념 증명 (Proof of Concept; PoC)
- 특정 질문만 응답하면 되는 문답형 에이전트
- 도구가 필요 없는 경우
- 실제 운영하고자 하는 것이 아닌, 실험 및 배움이 목적인 경우

### 설계 프레임워크 사용을 권장하는 경우

아래에 해당한다면 설계 프레임워크 사용을 적극 권장합니다:

- 엔터프라이즈 데이터 또는 민감한 데이터에 접근
- 단순히 질문에 답하는 것을 넘어 실체 작업을 수행
- 여러 팀 또는 이해관계자가 관여
- 보안, 컴플라이언스, 또는 거버넌스 요구사항이 적용
- 확장, 기능 향상, 또는 재사용될 것으로 기대
- 자율형 에이전트 구축
- 멀티 에이전트 시스템 구축
- 워크플로우 특화 에이전트 구축

### 설계 프레임워크 미사용 시 발생할 수 있는 문제

설계 없이 구축을 시작하면 아래와 같은 문제가 발생할 수 있습니다:

- ⚠️ 거버넌스 또는 보안 관련 문제가 뒤늦게 식별
- ⚠️ 에이전트의 상당 부분을 새롭게 구축 (재작업)
- ⚠️ 기술적으로 작동은 하지만, 비즈니스 목적을 달성하지 못하는 에이전트
- ⚠️ 안정적으로 확장하지 않는 연약한 솔루션

## 구성 요소 및 작성 방법

### 🎯 Goal (목표)

> "어떤 결과를 달성하려 하는가?"를 먼저 묻는다. 도구나 커넥터가 아닌 **결과(outcome)** 중심으로 작성한다.

**Jobs-To-Be-Done 형식으로 작성:**
- **As a** \<사용자\>
- **I need to** \<해야 할 일\>
- **So that** \<얻고자 하는 결과\>

**예시:** "As a new employee, I need to understand my local HR policies so that I can navigate onboarding confidently."

⚠️ **흔한 실수:** 결과 대신 기능부터 설계, 엣지 케이스 위주 설계, 성공 기준 미정의

---

### ⚡ Triggers (트리거)

에이전트가 작동을 시작하게 하는 이벤트, 조건, 입력을 정의한다.

**예시:**
- 채팅에서 사용자 메시지 수신
- 공유 받은 편지함에 새 이메일 도착
- 시스템에 새 레코드 생성
- 예약된 반복 작업

⚠️ **흔한 실수:** 트리거 미정의, 불예측한 사용자 행동에 의존, 필요한 컨텍스트 누락

---

### 🔧 Tools and Integrations (도구 및 연동)

에이전트가 "알고 있는 것"이 아닌 "할 수 있는 것"을 정의한다.

**예시:**
- ServiceNow 커넥터 → 티켓 상세 조회
- Microsoft Entra ID → 사용자 위치 확인
- Jira API → 작업 항목 업데이트
- Outlook 커넥터 → 이메일 회신

⚠️ **흔한 실수:** 작업 로그 미기록, API 안정성 과신, 과도한 권한 부여, 오류 시 폴백 미정의

---

### 📡 Channels (채널)

에이전트가 배포되어 사용자와 상호작용하는 플랫폼/인터페이스를 정의한다.

**예시:**
- Microsoft Teams
- SharePoint
- Microsoft 365 Copilot
- 웹 채팅 또는 음성 인터페이스

⚠️ **흔한 실수:** 사용자가 실제로 일하는 곳이 아닌 배포 편의성 기준으로 채널 선택, 채널별 제약 무시

---

### 🧠 Knowledge and Data (지식 및 데이터)

에이전트가 추론에 사용할 정보와 해당 데이터의 위치를 문서화한다. 데이터 품질, 최신성, 권한 경계를 고려한다.

**예시:**
- 내부 문서, 데이터베이스
- 웹사이트, 지식 베이스
- 내부/외부 시스템

⚠️ **흔한 실수:** 데이터 거버넌스 미흡, 문서를 "지식"으로 혼동, 지식 소스 간 충돌, 권한/접근 제어 미설계

---

### 🔀 Flows and Orchestration (플로우 및 오케스트레이션)

에이전트 내에서 작업이 구조화되고 순서화되는 방식을 정의한다. 결정적 플로우/토픽 사용 시점, 오케스트레이션 의존 시점, 사람 개입 시점을 설계한다.

**플로우/토픽이 필요한 경우:**
- 다단계 데이터 수집
- 가이드 기반 문제 해결 또는 의사결정 트리
- 컴플라이언스/정책 기반 프로세스
- 고위험 또는 되돌릴 수 없는 작업

⚠️ **흔한 실수:** 플로우 과잉 구조화(경직), 플로우 부재(예측 불가), 인간-에이전트 책임 경계 불명확

---

### 📋 Instructions and Behavior (지침 및 행동)

에이전트의 역할, 추론 방식, 응답 방식, 도구/지식 사용 방식, 어조와 안전 규칙을 정의한다.

**예시:**
- 역할: "You're the IT Email Support Agent responsible for reading incoming mailbox messages..."
- 순서화된 행동: "Step 1: 지식 베이스 확인 → Step 2: ServiceNow 도구 호출 → Step 3: 정보 없으면 에스컬레이션"
- 실패 처리: "지식이 없거나 도구 호출 실패 시 추측하지 말고, 명확한 한계와 다음 단계를 안내"

⚠️ **흔한 실수:** 지침이 너무 모호함, 도구/지식/에이전트 사용 기준 미정의, 상충하는 지침

---

### 🏗️ Agent Architecture and Components (에이전트 아키텍처)

여러 에이전트를 사용해야 하는 경우를 정의한다:
- 도메인이 크거나 구분됨
- 팀별 소유권이 다름
- 권한/접근 수준이 다름
- 특화된 추론이 필요함

**예시:**
- 메인 에이전트 → IT 에이전트에 티켓 조회 위임
- 지식 에이전트가 문서 Q&A 담당
- 라우팅 에이전트가 전문 에이전트로 분기

⚠️ **흔한 실수:** 과도한 위임(아키텍처 난립), 단일 거대 에이전트, 위임 경계 미정의

---

### 🔐 Governance & Risk Management (거버넌스 및 위험 관리)

에이전트가 책임감 있고 안전하며 예측 가능하게 작동하도록 거버넌스, 보안, 모니터링 방식을 정의한다.

**포함 사항:**
- 인증 및 접근 모델
- 작업 권한 및 가드레일
- 안전 및 콘텐츠 보호
- 로깅, 감사, 추적 가능성
- 운영 소유권

⚠️ **흔한 실수:** 거버넌스 설계를 너무 늦게 시작, 과도한 권한 부여, AI 책임 원칙 미통합

---

### 📊 Evaluation and Optimization (평가 및 최적화)

실제 시나리오를 시뮬레이션하는 테스트를 정의하여 에이전트의 정확도, 관련성, 응답 품질을 측정한다.

**측정 항목:**
- 정확도 및 관련성
- 절약된 시간 또는 효율성
- 도입률 및 사용량
- 만족도 및 신뢰 지표
- 잘못된 정보 감지

⚠️ **흔한 실수:** 배포 후 평가(너무 늦음), 기준선 없음, 실제 시나리오 미반영, 회귀 감지 없음
